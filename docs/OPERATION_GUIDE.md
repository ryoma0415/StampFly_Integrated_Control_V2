# StampFly Integrated Control — 段階的安全飛行手順

本書はオペレータ向けの飛行運用手順。セットアップと UI の詳細は
[../README.md](../README.md)、フェイルセーフの正典は
[PROTOCOL.md](PROTOCOL.md)「タイミング・フェイルセーフ(規範)」。
v2 の運用手順(モーター実験モード §11、ヨー較正・FF パイプライン §12、
ヨー角制御飛行 §13、円軌道モード §14、飛行後解析 §15)は本書後半にある。

**原則: 各フェーズの確認項目がすべて合格するまで、次のフェーズへ進まない。**
リスクの低い順(机上 → 機体電源 OFF での通信確認 → 地上確認 → 短時間ホバー →
本飛行)に進める。

---

## 1. 飛行前チェックリスト

- [ ] 2m×2m 以上の空間を確保(Position モードは MoCap 計測範囲内)
- [ ] 周囲に人・ペットがいない。壊れやすい物を片付けた
- [ ] バッテリー充電済み(UI の電圧表示で 3.5V 以上が目安。3.4V 未満は飛行不可)
- [ ] プロペラの損傷なし。プロペラガード装着(推奨)
- [ ] 機体は平らな床の中央に水平に置く
- [ ] PC のキーボードに手が届く位置にいる(**SPACE = 緊急停止**)
- [ ] 飛行は **release ビルド**のファームで行う(README §3)

## 2. 緊急時の操作(最初に覚える)

下から順にエスカレーションする。**迷ったらまず SPACE**。

| レベル | 操作 | 効果 |
|---|---|---|
| 1 | UI の **STOP** ボタン(どちらのタブにもある) | CMD_STOP 送信 → 即時着陸。**全飛行状態で受理**(TAKEOFF/LANDING 中含む)。MOTOR_TEST 中はモーター停止 → WAIT |
| 2 | **SPACE キー** | どこからでも(フォーカス位置に関わらず)即 STOP。**全タブで有効**。Experiment 中は CMD_MOTOR_STOP も送出し、スイープ/シーケンスも中断する。連打してよい |
| 3 | **機体の電源スイッチ OFF** | 最終手段。その高度から落下するため、低高度で機体に安全に近づける場合のみ |

補足(知っておくと落ち着いて対処できる):

- STOP 送信後 600ms 以内に着陸イベントがなければ、pc_server が**自動で再送**する
  (最大3回)。「応答がありません」警告が出ても機体側フェイルセーフが控えている。
- **PC・サーバ・リレーのどれが落ちても機体は自分で着陸する**: 飛行中に
  CMD_SETPOINT(50Hz ハートビート)が 200ms 途絶えると水平化、500ms で
  自律着陸(link_loss)に入る(§9 の表)。USB を抜く・サーバを強制終了する
  だけでも結果的に着陸する。
- OverG(衝突など >2.0g)を検出するとモータ即停止 → COMPLETE になる。復帰は
  機体を回収・静置してから UI の **Re-arm (RESET)**(推定高度 0.15m 未満でのみ受理)。

## 3. フェーズ0: 机上検証(リスク: なし、ハード不要)

```sh
# プロトコル実装の一致テスト
cd protocol && python3 -m pytest tests/ -q          # 全件 pass

# サーバのロジックテスト(フェイク serial/NatNet)
cd pc_server && .venv/bin/python -m pytest tests/ -q # 全件 pass

# ファームのコンパイル確認
~/.platformio/penv/bin/pio run -d firmware_stampfly -e release
~/.platformio/penv/bin/pio run -d firmware_relay -e release
```

すべて成功すること。venv が無ければ README §4.1 を先に実施する。

## 4. フェーズ1: 通信確認 — 機体電源 OFF(リスク: なし)

**機体の電源は OFF のまま**、PC⇔リレー区間だけを確認する。

1. リレーを USB で PC に接続し、pc_server を起動(README §5)、UI を開く。
2. ヘッダで機体プロファイルを選択し、リレーのポートを選んで「接続」。

確認項目:

- [ ] **シリアル**インジケータが緑
- [ ] **リレー**インジケータが緑(機体が OFF でも RLY_SET_TARGET の ACK は返る。
      黄のままなら README §7 のリレー欄を確認)
- [ ] コンソールに「リレーターゲット設定完了: <MAC> ch<N>」が出る
- [ ] **機体**インジケータは消灯のまま(電源 OFF なので正常)
- [ ] リレー統計 `relay ↑.. ↓.. err:0` が毎秒更新される(err が増えないこと)
- [ ] **STOP ボタンと SPACE キーを実際に押す**: コンソールに「CMD_STOP 送信」、
      しばらくして「CMD_STOP への応答がありません(再送上限到達)」警告
      —— 機体 OFF ではこれが**期待どおりの動作**。緊急停止の操作を体に覚えさせる

## 5. フェーズ2: 機体地上確認 — プロペラ回転なし(リスク: 低)

1. 機体を平らな床に置き、電源 ON。INIT → CALIBRATION → WAIT と進む間
   (数秒+静定 3 秒)は**機体を動かさない**(ジャイロ較正と AHRS リセット)。
2. UI で確認:

- [ ] **機体**インジケータが緑、状態バッジが **WAIT — 待機**
- [ ] 電圧が表示され 3.5V 以上(黄/赤なら充電してから)
- [ ] レイテンシが数値表示される(セットポイント往復が機能している証拠)
- [ ] 機体を**手で**ゆっくり傾けると姿勢バー(Roll/Pitch)が追従する
- [ ] 高度スライダを動かすと「高度(現在 vs 目標)」の目標マーカーが追従する
      (高度は離陸前から変更可能。これが離陸目標高度になる)

## 6. フェーズ3: 短時間ホバー(リスク: 中)

§1 のチェックリストを完了してから。**Posture タブ**で行う。

1. 高度スライダを 0.3m 程度に設定。Roll/Pitch はスライダ中央(0°)のまま。
2. ログ保存トグルを ON(後の較正・解析のため推奨)。
3. **START** → 確認ダイアログで OK → **1〜2 秒で STOP**(または SPACE)。

- [ ] まっすぐ離陸し、STOP で降下して着陸する
- [ ] **流れた方向(前後左右)をメモする**(§7 の較正に使う)
- [ ] 異常(片側に強く倒れる・異音)があれば即 SPACE → 電源 OFF し、
      プロペラ・モータを点検

問題なければ **5〜10 秒の通常ホバー**を行い、安定性とドリフト方向を再確認する。
複数回の離着陸(START→STOP→START→STOP)も正常に動くこと。

## 7. 機体プロファイル較正(角度バイアス)

機体には個体差があり、指令 0° でも一方向に流れる(ドリフトする)ことがある。
このドリフトを **PC 側プロファイルのバイアス**(`pc_server/config/airframes.json`
の `roll_bias_deg` / `pitch_bias_deg`)で打ち消す。ファームには入れない。

**考え方(ドリフト方向からの逆算)**: ホバー中に流れる方向 = 機体が実際に
傾いている方向。これを打ち消すには、**流れる方向と逆向きに傾ける指令**
(=バイアス)を常時加算する。必要量は流れの速さに応じて 0.5°〜2° 程度が目安。

**手順(Posture モードのスライダで「測る」)**:

1. バイアス 0 のプロファイルで 0.3m ホバーし、ドリフト方向と速さを観察する
   (5 秒程度。ログ ON 推奨)。
2. ホバーを続けたまま、**流れと逆方向に Roll/Pitch スライダを 0.5° 刻みで動かし、
   ドリフトが止まる値を探す**。符号に自信がなければ小さく振ってみて、
   悪化したら逆へ。バッテリー残量に注意し、必要なら着陸して再開する。
3. ドリフトが止まったときの**スライダ値がそのままバイアス値**。値を読み取って
   着陸(STOP)する。
4. UI ヘッダの**「編集」**で機体プロファイル編集を開き、該当プロファイルの
   `roll_bias_deg` / `pitch_bias_deg` にスライダ値を記入して**保存**
   (例: `-1.0` / `0.5`)。`notes` に測定日を残す。サーバの再起動は不要 ——
   保存と同時に選択中プロファイルのバイアスが再適用される(飛行中は
   選択中プロファイルの変更が拒否されるため、**必ず着陸後に保存**する)。
5. スライダ中央(0°)で再ホバーし、その場に留まることを確認。残るようなら
   1°→0.5°→0.25° と刻みを細かくして繰り返す。

検証・微調整のヒント:

- ログの `roll_ref_deg`/`pitch_ref_deg` は**バイアス加算後**の送信値、
  `roll_bias_deg`/`pitch_bias_deg` は適用中バイアス(LOG_STRUCTURE.md)。
  引き算すればスライダ操作分が復元できる。
- Position モード運用後の上級者向け確認: 安定ホバー中のログの `pid_x_i`/`pid_y_i`
  (XY PID の I 項)が大きな定常値を持つ場合、それは XY PID がバイアス相当を
  肩代わりしている兆候。その分をプロファイルへ移すと I 項の負担が減る
  (符号は座標変換を挟むため、移した後に必ず実飛行で確認する)。

## 8. フェーズ4: 本飛行

### 8.1 Posture モード(手動の姿勢・高度指令)

- Roll/Pitch スライダは飛行中のみ操作可。**小さく動かして必ず中央に戻す**
  (「中央に戻す」ボタン)。サーバ側でスルーレート 30°/s に制限されるが、
  指令を出しっぱなしにすれば機体は流れ続ける。
- 高度スライダは 0.1–1.0m。変更は 0.3m/s に制限されて反映される。
- 着陸は STOP。最大飛行時間 120 秒で自動着陸する(§9)。

### 8.2 Position モード(MoCap による位置保持)

前提: Motive/NatNet 設定済み(README §4.3)。

1. **離陸前に** Position タブで確認:
   - [ ] MoCap インジケータが「受信中」(緑)
   - [ ] 座標表示と XY プロットが、機体を手で動かすと追従する
   - [ ] 値が制御座標系として妥当(原点・軸方向が想定どおり)
2. 目標を「**この場で**」プリセットにする(現在位置 XY を目標化。
   離陸直後の横移動を防ぐ)。目標 Z は 0.3m 程度から。
3. **START**(MoCap が新鮮でないと拒否される)。
4. 安定したら目標 X/Y を **0.1〜0.2m 刻み**で動かして追従を確認。
   「原点」プリセットで (0,0) へ戻せる。
5. 着陸は STOP。MoCap が 300ms 途絶えると水平固定+警告、2 秒で自動着陸する(§9)。

### 8.3 終了手順

1. STOP → 着陸と WAIT 復帰を確認
2. ログ保存トグル OFF(ファイルが閉じられる)
3. UI で「切断」→ 機体の電源 OFF → バッテリーを外して保管

## 9. フェイルセーフ一覧(PROTOCOL.md より転記)

正典は [PROTOCOL.md](PROTOCOL.md)。PC 側閾値の実効値は
`pc_server/config/server.json`(`failsafe` 節)、ファーム側は
`firmware_stampfly/src/config.hpp`。

| 条件 | 動作 | 実装場所 |
|---|---|---|
| CMD_SETPOINT 途絶 >200ms(飛行中) | roll/pitch を水平(バイアス込み0)へ、alt_ref 維持。**ヨー制御中なら途絶時点の推定ヨー角をラッチして保持**(v2) | ファーム |
| CMD_SETPOINT 途絶 >500ms(飛行中) | LANDING へ遷移(reason=8 link_loss)。着陸中のヨーはレートダンピングのみ | ファーム |
| EKF 不健全(磁気更新凍結 ffg bit5 / アンカー無効 / FF無効)で est_mode=EKF(v2) | ヨー角制御を止めてレートダンピングに縮退。ffg / ff_status で UI に通知(§13) | ファーム |
| CMD_MOTOR_RUN 途絶 >1.5s(MOTOR_TEST 中)(v2) | モーター自動停止 | ファーム |
| 低電圧(<3.34V、20Hz×5サンプル連続 ≈0.25s) | 飛行中: LANDING(reason=4)。WAIT: START拒否(reason=5) | ファーム |
| OverG(>2.0g) | モータ即停止 → COMPLETE(reason=7)。CMD_RESETでのみ復帰 | ファーム |
| 最大飛行時間 120s | LANDING(reason=3) | ファーム |
| MoCap 途絶 >300ms(Positionモード) | setpoint を水平に固定+UI警告(円軌道中は軌道位相・接線ヨー目標も凍結) | pc_server |
| MoCap 途絶 >2s(Positionモード) | CMD_STOP 送信(自動着陸)。円軌道中も同一 | pc_server |
| STOP 送信後 600ms 以内に LANDING/WAIT イベントなし | CMD_STOP 再送(最大3回)+UI警告 | pc_server |
| シリアル切断 | UI赤色警告(機体側は上記リンク喪失で自律着陸。MOTOR_TEST 中は CMD_MOTOR_RUN 途絶停止) | pc_server |

このほか UI/サーバ/ファームの三層で指令クランプがかかる
(UI ±5°・0.1–1.0m・ヨー±180° / サーバ ±10°+30°/s・0.1–1.2m+0.3m/s・
ヨー±180°+45°/s / ファーム ±30°・0.05–1.5m・ヨーレート±1.0rad/s。
ARCHITECTURE.md「安全クランプ(多層)」)。

## 10. よくある問題(飛行時)

| 症状 | 対処 |
|---|---|
| START しても離陸しない | 状態バッジを確認。WAIT 以外では受理されない。「離陸拒否: START_REJECTED_…」がコンソールに出ていれば理由(低電圧/未準備)を解消。実験モード中は離陸不可(§11) |
| ホバーが一方向に流れる | §7 のバイアス較正。較正済みなら床の傾き・プロペラ損傷・気流を疑う |
| 飛行中に勝手に着陸した | コンソールのイベント(reason)を確認: link_loss=電波/USB、low_voltage=充電、max_flight_time=仕様(120s) |
| COMPLETE から動かない | OverG 後。機体を回収・静置(高度 0.15m 未満)して Re-arm (RESET) |
| Position で警告「MoCap 途絶」 | マーカーの隠れ・計測範囲外。頻発するなら Motive のカメラ配置と `control.json` のフィルタ設定を見直す |

# v2 運用手順(実験モード・ヨー較正・ヨー制御飛行・円軌道・飛行後解析)

## 11. モーター実験モード(Experiment)の安全手順

Experiment タブは**ベンチ実験専用**で、機体はプロペラを回したまま飛ばない
(MOTOR_TEST 状態。飛行制御 PID・ミキサは動かない)。プロペラ回転を伴うため、
**機体の固定が最優先の安全策**である。

手順(この順番を守る):

1. **機体を固定する**: プロペラを外せない実験(スイープ)は、機体を
   テープ等で作業台へ確実に固定する。周囲に紙・軽量物を置かない。
2. 機体電源 ON → WAIT を確認し、UI で接続する。
3. **Experiment タブへ切替**: pc_server が 50Hz セットポイント送信を停止し、
   CMD_MODE(1) を送って機体を MOTOR_TEST に遷移させる。バッジが「有効」に
   なることを確認(「未有効」のままなら「実験モードを有効化」を押す。
   機体が WAIT 以外だと ACK bad_state で拒否される)。
4. **「機体をテープで確実に固定したことを確認した」チェックを ON** にする
   (これを入れるまでモーター/スイープ/シーケンスの開始ボタンは無効)。
5. モーターテスト: duty ボタン(0.1〜1.0)を選んで Start。
   **duty 0.6 以上は「高出力許可」チェックが別途必要**(機体固定を再確認)。
   duty はソフトスタート 2.0 duty/s でランプする。
6. 停止は **Stop ボタンまたは SPACE**(Experiment 中の SPACE は
   CMD_MOTOR_STOP+CMD_STOP を送出)。

多層の自動保護(操作ミス・リンク断に対する備え):

- PC は CMD_MOTOR_RUN を 0.4s 周期で再送(キープアライブ)し、機体は
  **1.5s 途絶でモーターを自動停止**する(USB を抜いても止まる)。
- スイープは **電池電圧 <3.0V(過放電防止)/ 電流 >12A / TLM_EXP 2s 途絶**で
  自動中断する(収集済みデータは保存される)。
- 加算性シーケンスは各スイープ前に**電池ガード(vbat ≥ 3.5V、UI で変更可)**で
  ブロックし、不足時は「電池交換 → 再開」の明示操作を要求する。
  スイープ間に冷却 10s を挟む。
- 実験を終えて Posture/Position タブへ戻ると、モーター停止確認 →
  CMD_MODE(0) → WAIT 復帰 → 50Hz 送信再開、の順で自動処理される。

## 12. ヨー較正・FF パイプライン(一連の手順)

ヨー角制御飛行(§13)の前提となる較正の全手順。詳細仕様は
[FF_PIPELINE.md](FF_PIPELINE.md)。すべて Experiment タブで行う(§11 の
安全手順を先に完了しておく)。

1. **地磁気プロファイル**: 「地磁気」パネルで都道府県を選択 → 保存+機体へ
   CMD_GEOMAG_SET 送信される。
2. **加速度6面較正(推奨)**: 「Accel6」パネルで6面(x±/y±/z±)を各面静置して
   キャプチャ(約1秒平均)→ 適用(適用時に姿勢参照がリセットされる)。
   必要なら「Attitude 0」で治具上のマウントオフセットも設定する。
3. **3D磁気較正**: 「3D磁気キャリブレーション」開始 → **機体をあらゆる向きに
   ゆっくり回す**(最低 80 サンプル、上限 6000)→ 停止 → Fit → 適用。
   適用で機体 NVS に書き込まれ、PC 側にも `pc_server/config/mag3d_calibration.json`
   として保存される(以後のスイープ CSV の b*_cor 計算に使用)。
   **注意: mag3d を適用/クリアすると機体側で FF は自動無効化(ff_mode=0)される**。
   FF 適用済みなら再抽出・再適用が必要。
4. **ヨーゼロ(任意)**: 現在の機首方位を 0° 基準にするなら「Yaw 0」。
   **FF が有効(ff_mode≠0)だと拒否される**ため、FF 適用前かここで済ませる
   (または一時的に FF モードを off にする)。
5. **スイープ 8 本**: FF 抽出には次の 8 本(16 ファイル)が必要。
   - **全機(FL+FR+RL+RR)× 4 姿勢**: 機体の向きを変えて 4 本。各ランの
     notes の「方位」に `Yaw=0°` / `Yaw=90°` / `Yaw=±180°` / `Yaw=-90°` のように
     **4 種の異なる姿勢**を必ず記録する(抽出時に検証される)。
   - **単機 FL / FR / RL / RR × 各 1 本**: 加算性シーケンス(自動 4 本)を使うと
     電池ガード・冷却込みで実行できる。
   - パターンは往復(updown、既定)推奨。中断ラン(aborted)は抽出に使えない。
6. **フォルダ整理**: `pc_server/data/sweep_results/` の直下にサブフォルダ
   (例 `DroneTest_20260629/`)を作り、8 本ぶんの
   `sweep_*_samples.csv` + `sweep_*_meta.json` を移動する(UI の抽出は
   フォルダ選択式)。
7. **FF 抽出**: 「FFプロファイル」パネルでフォルダを選び「抽出実行」
   (内部で `data_analysis/make_ff_profile.py` をサブプロセス実行、120s
   タイムアウト)。警告(mag3d 不一致等)が出たら内容を確認する。
   CLI で直接実行してもよい:
   `cd data_analysis && .venv/bin/python make_ff_profile.py --folder <フォルダ名>`。
8. **FF 適用**: プロファイルを選び、ff=方式B・est=EKF(既定)で「適用」。
   CMD_FF_BEGIN→LUT×n→MOT×4→AUX→COMMIT(CRC-32 照合)の分割転送と
   読み戻し検証が自動で行われ、成功すると「適用・CRC照合OK」+適用中バナーに
   なる。**「mag3d が取得時と一致しません」警告が出た場合は原則スイープ再取得**
   (force 適用は係数が現在の磁気較正空間と不整合になることを理解した上で)。
9. **ヨー評価**: §13 の Phase 0(ヨー制御 OFF 飛行)でログを取り、§15 の
   flight_log_viewer で 4 系統比較・EKF 診断を確認する。

## 13. ヨー角制御飛行の段階手順

ヨー角制御(CMD_SETPOINT flags bit1)は v2 の新機能であり、
**必ず OFF での検証飛行(Phase 0)から始める**。

### Phase 0: ヨー制御 OFF で飛行し、推定の符号と健全性を確認する

1. ヨー角制御トグルは **OFF のまま**(機体は V1 と同一のレートダンピングのみ)、
   ログ保存 ON で通常のホバー飛行を行う(§6/§8)。飛行中に機体を手で回す必要は
   ない — 離着陸時の自然なヨー運動で十分。可能なら Position モードで飛ぶと
   MoCap ヨー真値も同時記録される。
2. 着陸後、flight_log_viewer(§15)で **ヨー4系統比較**
   (`10_yaw_four_sources.png`)を確認:
   - `tlm_yaw_rad`(Madgwick)/ `tlm_yaw_est_rad`(EKF)/
     `tlm_yaw_gyro_int_rad`(ジャイロ積算)/ `mocap_yaw_deg`(MoCap 真値、
     Position のみ)の **4 系統が同じ向き・同じ量で動いている**こと
     (符号が逆の系統があれば較正・設定を疑い、ヨー制御 ON に進まない)。
   - EKF 診断(`12_ekf_diagnostics.png`)で NIS が定常(概ね <5.99)で、
     ゲート発火(ffg)が散発以上に出ていないこと。ドリフト率 [°/min] が
     ジャイロ積算より十分小さいこと。
3. 飛行中は UI のヨー推定モニタ(Madgwick / EKF / ジャイロ積算の数値と
   EKF 健全性バッジ)でも大まかな一致を確認できる。

### Phase 1: ヨー角制御 ON(低ゲインから)

1. FF プロファイル適用済み(§12)・EKF バッジが「EKF OK」であることを確認し、
   ホバー安定後に**ヨー角制御トグルを ON** にする(ON 時に FF 未適用だと
   「FF未適用」警告バッジが出る。飛行は可能だがモーター電流による磁気干渉を
   受けるため推奨しない)。
2. ヨースライダを **±10〜20° だけ**動かし、行き過ぎ・振動なく回頭して
   保持することを確認する。指令はサーバ側で最短経路 45°/s に整形され、
   機体側はヨーレート ±1.0 rad/s にクランプされる。
3. 問題があれば即着陸。ヨー角 PID の初期ゲイン
   (`firmware_stampfly/src/config.hpp` の `yaw_angle`: kp=3.0, ki=0.3
   (ti=10.0 表現), kd=0.0)は**未飛行の初期値(要飛行チューニング)**であり、
   振動するなら kp を下げて再ビルドする。
4. 徐々に大きな回頭(±90°、±180°)へ進める。±180° 指令は wrap により
   どちら回りにもなり得ることに留意する。

### EKF 不健全時の縮退挙動(知っておくこと)

- est=EKF で飛行中に EKF が不健全(磁気更新凍結 ffg bit5 / アンカー無効 /
  FF 無効)になると、機体は**ヨー角制御を自動で止めてレートダンピングに縮退**
  する(飛行中のヨーソース切替による指令段差を作らないため)。機体は落ちないが
  **ヨーは保持されなくなる**(ゆっくり流れる)。UI の EKF バッジが「EKF注意」に
  変わり、ffg / FF状態表示で原因を確認できる。
- 縮退が起きたら回頭操作をやめ、必要なら着陸する。回復は着陸後に
  アンカー再取得(モーター停止中のみ受理)や FF 再適用で行う。
- CMD_SETPOINT 途絶(>200ms)時は、ヨー制御中なら**途絶時点の推定ヨー角を
  ラッチして保持**する(0° への回頭はしない)。>500ms で自律着陸(既存)。

## 14. 円軌道モードの運用(Position)

1. 前提: Position モードで安定したホバリング(固定目標)ができていること。
2. 軌道セレクタで「円軌道」を選び、パラメータを入力する
   (制限は `control.json` の trajectory 節):
   中心 X/Y(±2.0m 以内)・半径(0.05–1.5m)・周期(3–120s)・
   回転方向(CW/CCW)・高度。初回は**半径 0.3m 程度・周期 20s 程度**の
   ゆっくりした円から。
3. **「円軌道開始」**: 現在位置から円周上の最近傍点へ滑らかに合流し、位相は
   その点から進む(MoCap が新鮮(0.3s 以内)でないと開始は拒否される)。
   XY プロットに目標軌道の円が重畳表示される。
4. **「停止(ホバ復帰)」**: その時点の軌道目標でホバリングに戻る(その場停止)。
   緊急時は通常どおり STOP / SPACE(即着陸)。
5. **「進行方向を向く」(face_tangent)**: ヨー角制御 ON のときのみ有効で、
   yaw_ref を接線方向に追従させる。接線ヨーの角速度は 360°/周期 であり、
   ヨースルーレート上限(45°/s)を超えないよう **周期 8 秒以上が必要**
   (8s 未満は開始が拒否される)。§13 Phase 1 を終えてから使うこと。
6. MoCap 途絶時は既存フェイルセーフと同一(300ms で水平化+軌道位相を凍結、
   2s で自動 STOP)。復帰時は位相が凍結時間ぶん補正され連続に進む。

## 15. 飛行後解析(flight_log_viewer)

1. 飛行時にログ保存トグルを ON にしておく(50Hz・94列。列定義は
   [LOG_STRUCTURE.md](LOG_STRUCTURE.md))。着陸後にトグル OFF で
   ファイルが閉じられる。
2. 解析(初回は `flight_log_viewer/README.md` のセットアップで venv を作る):

   ```sh
   cd flight_log_viewer

   # 対話モード(出力内容とログを選ぶ)
   .venv/bin/python visualize.py

   # バッチ: 静止画グラフ+ヨー解析+サマリレポート
   .venv/bin/python visualize.py ../logs/20260706_120000_position.csv

   # アニメーション MP4(動画なし)/ すべて生成
   .venv/bin/python visualize.py ../logs/xxx.csv --animation
   .venv/bin/python visualize.py ../logs/xxx.csv --all

   # 2 ログのヨー安定性比較(FF あり/なし等の対照)
   .venv/bin/python visualize.py ../logs/a.csv --compare ../logs/b.csv
   ```

3. `output/<ログ名>/` に静止画 01〜14・`summary.txt`・`index.html`
   (統計+全グラフの HTML レポート)が生成される。ヨー評価で見るのは主に:
   - `10_yaw_four_sources.png` — ヨー4系統比較(§13 Phase 0 の判定)
   - `11_yaw_error.png` — 対 MoCap ヨー誤差+RMS/ドリフト率 [°/min]
   - `12_ekf_diagnostics.png` — NIS・b_m・ΔB̂・ffg ゲートタイムライン
   - `14_yaw_tracking.png` — ヨー指令追従(PC 指令/機体適用目標/実測)
4. スマホ動画と同期した合成アニメーションを作る場合は「ログ記録開始と同時に
   録画開始」しておき、`--video <動画> --track` を付ける(opencv-python 必要)。
