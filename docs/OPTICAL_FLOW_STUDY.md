# オプティカルフロー(PMW3901)活用検討書

作成日: 2026-07-06
位置づけ: **将来検討のための調査・設計文書**(現行 V2 の実装仕様ではない。実装仕様の正典は
PROTOCOL.md / ARCHITECTURE.md)。ヨー角制御プロジェクトの完了後に、StampFly 底面搭載の
オプティカルフローセンサー PMW3901 を追加活用する価値があるかを検討した結果をまとめる。

出典: `../Yaw_Estimation_Project/StampFly_technical_specification/` の各仕様文書、
V2 実コード、および外部事例(Bitcraze Crazyflie Flow deck v2 / PX4 / ArduPilot の実装)。

---

## 結論(要約)

1. **(a) 取得できる値**: フレーム間の画像移動量カウント(ΔX/ΔY、16bit 符号付き・無次元)と
   品質指標(SQUAL・シャッター値・生画像統計)。物理量(m/s や rad/s)は出力されない。
2. **(b) 生値のまま使えるか**: **使えない**。磁気センサーと同程度に多段の処理ロジック
   (①スケール実機校正 → ②ジャイロ回転補償 → ③高度スケーリング → ④チルト補正 →
   ⑤品質ゲート → ⑥カルマンフィルタ融合)が必須。特に②を省くと制御ループ内で正帰還が
   生じ発振する。
3. **(c) 利用価値**: **ある。ただし用途を選ぶ**。最有力は「MoCap/リンク途絶時の
   開ループ水平化を、フロー速度によるゼロ制動に置換するフォールバック」で、効果が確実かつ
   平常時制御に触れないため低リスク。MoCap 稼働中の常時内側ループは改善幅が小さい可能性が
   高く優先度低。MoCap なし位置ホールドは「速度ホールド」であってドリフトは残る(期待値管理
   必須)。本環境は MoCap 真値とフロー速度を直接比較できるため、**Phase 0(ログのみ)の
   検証・校正が安価に高精度で行える**のが最大の強み。

## 1. (a) PMW3901 から取得できる値

### 1.1 センサー概要と接続

- PMW3901MB-TXQT。StampFly 底面に**下向きに実装済み**(追加ハードウェア不要)。
- 4線式 SPI(**上限 2MHz**)。**BMI270 と同一 SPI バスを共有**
  (MOSI=G14 / SCK=G44 / MISO=G43 共通、CS は BMI270=G46、**PMW3901=G12**)。
- V2 ファームは現在 PMW3901 を未使用だが、`imu_init()` が G12 を HIGH に待避しており
  ([imu.cpp:43](../firmware_stampfly/src/imu.cpp))、`spi_bus_add_device()` で
  第2デバイス(2MHz / mode3)を追加するだけで物理層は成立する。
- 旧ファームウェア(Previous_Version 配下)にも PMW3901 の使用コードは存在しない
  (grep で確認)。**ドライバは新規実装**となる(初期化手順は仕様文書に完備)。

### 1.2 取得できる生値

個別レジスタまたは **Motion_Burst(0x16)で 12 バイト一括読み**:

| 値 | レジスタ | 内容 | 単位/レンジ |
|---|---|---|---|
| Motion | 0x02 | 新規モーション検出フラグ(読むと Delta がフリーズ→読み出し順序遵守) | フラグ |
| **Delta_X / Delta_Y** | 0x03-0x06 | フレーム間の画像移動量 | **16bit 符号付きカウント(無次元)** |
| **SQUAL** | 0x07 | 表面品質(特徴量の多さ) | 8bit。**< 0x19 で低品質の目安** |
| Shutter_Upper/Lower | 0x0C/0x0D | シャッター値(露光) | **Upper==0x1F で飽和(照度不足)** |
| RawData_Sum/Max/Min | 0x08-0x0A | 生画像の統計 | 8bit |
| (デバッグ用)生画像 | 0x58/0x59 | 35×35px フレームキャプチャ | 飛行中使用不可(要フルリセット) |

### 1.3 光学仕様と制約(文書記載)

- 動作距離: **80mm〜無限遠**(最低高度 8cm 未満では保証外)
- 品質判定の唯一の定量基準: 「SQUAL < 0x19 **かつ** Shutter_Upper == 0x1F なら破棄」
- 最低照度 60 lux(データシート値)、最大検出角レート 7.4 rad/s
- **FOV・カウント→物理量の変換式は仕様文書に記載なし**(重要 — §2.1 参照)
- 初期化に PixArt 独自の「性能最適化レジスタ列」約80項目の書き込みが必要(手順は文書に完備)
- スケーリング用の対地距離は底面 VL53L3CX(mm 単位、約33ms 周期、0〜3000mm)から取得可能

## 2. (b) 生値のまま使えるか → 使えない。必須の処理チェーン

ヨー角推定(mag 生値 → mag3D 較正 → 電流FF補正 → EKF)と**完全に同型**の多段処理が必要。

| 段 | 処理 | ヨー推定での対応物 | 省略した場合の症状 |
|---|---|---|---|
| ① スケール実機校正 | Δcount → 角レート [rad/s]。係数は文書になく、**文献値も約 48.8〜4551 counts/rad と2桁乖離**(Crazyflie ≈48.8 / PX4 385 / ArduPilot ≈794 — レジスタ設定・単位定義依存) | 磁気スケール較正 | 制御ゲインが未知倍率で狂う。ダンピングが効かない or 発振。**校正なし投入は不可** |
| ② 回転成分補償 | フローから機体角速度 p,q 成分を減算(Crazyflie/PX4/ArduPilot 全実装が実施) | ジャイロとの融合 | **最重要**。姿勢指令→機体傾斜→フローに回転成分混入→「動いた」と誤検出→逆補正、の**制御ループ内正帰還で発振** |
| ③ 高度スケーリング | v = (flow_rate − body_rate) × h。h は既存 alt_kalman の Altitude2(400Hz 参照可) | — | 速度ゲインが高度に比例して変動(低空で過小・高空で過大) |
| ④ チルト補正 | h に cosφ·cosθ 補正(V2 の ToF 高度は傾き未補償のため) | 磁気のチルト補償 | 傾斜 20° で約 6% のスケール誤差。制動時(傾けて止める場面)に拡大 |
| ⑤ 品質・外れ値ゲート | SQUAL≥0x19、Shutter≠0x1F、Range0flag==0、0.08m≤h≤2.0m、\|Δ\| 上限(Crazyflie: 100count/10ms) | mag ノルムゲート → EKF 健全判定 | 低テクスチャ/光沢床・照度不足で無効データのまま制御に入り**機体が流される/暴れる** |
| ⑥ KF 融合 | 加速度で 400Hz predict、フローで ~100Hz update する 2状態速度KF(vx, vy) | ヨーEKF(gyro predict / mag update) | 生値微分の直接 FB は量子化ノイズでモータ指令が暴れ、強 LPF では位相遅れで効果減殺 |

①〜⑤は省略不可。⑥は低ゲインの制動用途なら軽い LPF でも成立し得るが、本格利用なら KF が定石。

**磁気センサーとの比較**: 磁気の「電流による系統外乱(FF 補正が必要)」に相当するものは
フローにはないが、代わりに「**床面・照度という環境依存の有効性問題**」がある。処理の重さは
同等、不確実性の性質が異なる(磁気=系統誤差の補正、フロー=有効/無効の判別)。

## 3. (c) V2 での利用価値の評価と制御ロジック案

### 3.1 用途別評価(価値の高い順)

#### a. MoCap/リンク途絶時のフォールバック制動 ★最有力・効果確実

現行のフェイルセーフは「setpoint 途絶 >200ms で roll/pitch=0 の水平化」「AUTO_LANDING 中
roll/pitch=0 固定」だが、これは**開ループであり既存の水平速度は保存されたまま流される**。
これをフロー速度による「速度ゼロ制動」(−Kv·v を角度指令に、±5〜8° クランプ)に置換すれば、
途絶時・自動着陸中の流され距離を大幅に削減できる。

- 効果: 高・物理的に確実。**平常時の制御には一切触れない**ため低リスク
- フロー不健全時は現行の水平化へ自動縮退(ヨーの EKF 不健全縮退と同型の設計)
- >500ms 自動着陸 / >2s STOP の階層は**残す**(置換ではなく改善)
- 挿入点は [flight_control.cpp](../firmware_stampfly/src/flight_control.cpp) の途絶分岐と
  auto_landing_step の 2 箇所のみ

#### b. 離着陸時の水平ドリフト抑制 ○(a とセット)

着陸降下中の抑制は a に包含され効果確実。離陸側は動作保証外の h<80mm 区間(最初の数 cm)は
原理的にカバー不能で、「h>10〜15cm 到達後に有効化」のヒステリシス付きゲートとなる。

#### c. MoCap なし位置ホールド(研究室外テスト飛行)○期待値管理必須

得られるのは Crazyflie Flow deck 相当の「その場ホバリング」であり、フローは相対速度のみの
ため**位置はデッドレコニングでじわじわ流れる**(Bitcraze 自身が明記)。定点保持ではない。
出張デモ・屋外相当テストの価値は本物だが、工数最大なので最後に。前提: 高度 2m 以下制限
(ToF スケーリング上限)、テクスチャある床、照度 60lux 以上。

#### d. MoCap 稼働中の常時速度内側ループ △優先度低(正直な評価)

MoCap 稼働中は PC 側が高精度の位置・速度を持つため、ノイズが大きくスケール誤差のある
フロー速度を常時混ぜる情報的メリットは小さい。改善が見込めるのは 50Hz setpoint 間+無線
レイテンシ区間の突風応答を 400Hz ローカルダンピングで埋める部分のみで、体感差は小さい
可能性が高い。**c への足場としてのみ正当化される**。

#### e. 研究データとしての価値(本環境固有)

MoCap 真値とフロー速度を同一ログで直接比較できる環境は貴重で、**Phase 0(ログのみ)だけでも
フロー速度推定の精度評価として成果になる**。ヨー統合で作った 94 列ログ+flight_log_viewer の
基盤がそのまま流用できる。

### 3.2 推奨統合アーキテクチャ(ヨー統合の前例踏襲)

実装場所は**機体側**(回転補償に 400Hz の p,q が必要+途絶時に PC なしで動くことが本質)。

```
firmware_stampfly/src/flow_estimation/        # yaw_estimation/ と同型の分離モジュール
  flow_config.hpp             # scale 係数・SQUAL 閾値・ゲート高度・ゲイン
  pmw3901_driver.{hpp,cpp}    # 仕様文書の初期化列+Motion_Burst 12B 読み(2MHz/mode3, CS=G12)
  flow_velocity_estimator.{hpp,cpp}  # 補償チェーン+2状態速度KF(vx, vy)
  flow_persistence.{hpp,cpp}  # scale/軸マッピングの NVS(飛行中書込禁止規約踏襲)
```

- **読み出し**: 100Hz(4tick 毎)、Motion_Burst ≈100µs/回。既存の位相スタガ(ToF tick・
  20Hz スロット tick では 1tick 繰延べ)の第3枠として組み込む。400Hz ループ予算への影響は
  該当 tick の 4% 程度で問題なし
- **推定器**: 状態 [vx, vy] の 2状態 KF。predict=400Hz(姿勢回転済み加速度)、
  update=100Hz(補償済みフロー速度)。観測ノイズはシャッター値または SQUAL 連動
  (PX4 の N_MIN〜N_MAX 補間方式)。既存 alt_kalman・ヨーEKF とは**疎結合を維持**
  (統合 EKF 化はデバッグ性の観点で初手ではやらない)
- **健全性ゲートと縮退**: flow_status ビット(ヨーの ff_status と同型)で PC・ログに常時
  公開し、不健全時はバンプレスに現行動作へ縮退
- **プロトコル/ログ**: TLM_STATE 135B→149B(vx, vy, 生 dx, dy, squal, flow_status の
  14B 末尾追加。上限 200B に対し余裕)。ログ列も末尾追加(LOG_STRUCTURE.md /
  viewer/constants.py / test_log_columns.py の 3 点更新パターン)

### 3.3 段階導入計画(ヨー統合と同じ流儀)

| Phase | 内容 | 判断基準 |
|---|---|---|
| **0** 読み取り+ログのみ | ドライバ実装、生 dx/dy/SQUAL をログへ。**地上純回転テスト**で軸マッピング・符号・scale 初期値を確定(純回転中はフロー=ジャイロ×scale になるはず — ArduPilot 校正と同原理)。**MoCap 併用飛行**でフロー速度 vs MoCap 微分速度を回帰し scale 精密化+床面適性判定 | 健全率 >90%、速度相関 >0.9、スケール残差 <10% |
| **1** 速度KF(制御未投入) | 2状態 KF 実装、推定値をログ→MoCap 速度と RMSE/遅延/遮蔽時挙動を評価 | 遮蔽時に安全に縮退すること |
| **2a** 途絶時フォールバック制動 | 途絶分岐と AUTO_LANDING を −Kv·v(クランプ付き)に置換。低ゲインから。意図的に setpoint 送信を止め流距離を現行比で計測 | 流距離の有意な削減 |
| **2b**(任意)常時ダンピング | Phase 2a の実測を見て go/no-go | — |
| **3**(任意)MoCap なしホバー | CMD_SETPOINT flags 空きビット or 新型で速度指令モード追加 | — |

### 3.4 やらない方がよいこと

1. 文献のスケール係数をそのまま使わない(2桁乖離。実機校正が唯一の確立手法)
2. MoCap 稼働中の位置制御をフローで置換・改善しようとしない(全指標で MoCap が上)
3. 位置ホールドのドリフトゼロを期待しない(デッドレコニング)
4. >500ms 着陸 / >2s STOP のフェイルセーフ階層を撤去しない(フローは床材・照度で無効化し得る)
5. 床面を過信しない: **単色・光沢床(リノリウム等)は文献上の最悪条件**。パターン周期の大きい
   カーペットも離陸時誤マッチ事例あり。Phase 0 で不適と出たら、テクスチャ付き離着陸パッドの
   敷設が制御チューニングより先
6. h<80mm でフローを使わない(動作保証外。離陸直後の数 cm は諦める)
7. フレームキャプチャモードを飛行中に使わない(モーション検知が止まりフルリセット必須)
